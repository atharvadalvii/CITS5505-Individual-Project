1. Introduction

While building this CITS5505 static site (tutorial page, JSON-driven quiz, and shared navigation), I used Cursor for in-editor suggestions and refactors, and ChatGPT for occasional longer explanations (for example, how fetch resolves paths and how localStorage behaves across reloads). AI did not replace testing in the browser or checking the Network tab when something failed silently.

Since the first draft, the tutorial page gained extra sections (HTTP, forms, DevTools), a table of contents with smooth scrolling, a reading mode that hides chrome for long study sessions, and a checklist whose ticks persist in localStorage under a dedicated key. The quiz bank grew to fifteen questions so the AJAX shuffle and pass logic are exercised on a wider set of misconceptions, not only the original three items.

2. Example 1: Loading and rendering quiz questions

Prompt used

“Write async JavaScript that fetches data/questions.json, stores the array in a variable, shuffles question order, then renders each question with radio options into the element with id quiz-container. Use one radio group per question named q0, q1, etc.”

Summary of AI output

The model produced a loadQuestions() function using fetch and response.json(), then looped with forEach to create blocks of <label> and <input type="radio"> tied to each question’s options array and the correct name attributes. It also suggested a simple shuffle helper.

Changes made to the code

  • Kept the fetch("data/questions.json") path but verified it in a local server context (opening the HTML file directly can break fetch); documented that for my own testing.
  • Aligned markup with the quiz page structure I already had: question blocks use classes such as quiz-card and quiz-question so the same CSS applies to AI-generated DOM.
  • Replaced the suggested shuffle with a Fisher–Yates implementation (see Example 2 / limitation) so order is not statistically biased.

Decision and verification

Decision: I accepted the fetch-and-render structure but rejected the random sort shuffle. Verification: I confirmed the Network tab returned JSON with HTTP 200 and that each rendered question used a unique radio group name. 

// Before (AI draft)
questions.sort(() => Math.random() - 0.5);

// After (implemented)
const shuffled = fisherYatesShuffle(questions);

3. Example 2: Highlighting unanswered questions on submit

Prompt used

“On quiz submit, if any question has no selected radio, don’t score yet. Add a visible error state to those question cards and scroll the first missing one into view.”

Summary of AI output

It recommended a CSS class (for example quiz-unanswered) with a red border and light background, clearing that class before each validation pass. The script walks each question index, checks querySelector('input[name="q' + index + '"]:checked'), toggles the class on the parent card, and calls scrollIntoView({ behavior: 'smooth', block: 'center' }) on the first invalid card.

Changes made to the code

  • Scoped selectors to the quiz container so the logic does not accidentally target radios elsewhere on the page.
  • Used classList.add / remove in one place after submit, instead of mixing inline style.border (easier to maintain next to the existing stylesheet rules).
  • Replaced a suggested alert() with an in-page message in #results-section (role="alert") so unanswered state is announced without blocking the UI; red card outline remains the primary visual cue.

Decision and verification

Decision: I accepted the unanswered-card pattern but rejected blocking alert(). Verification: I tested with partially answered submissions and confirmed that only missing cards were highlighted, the first missing question scrolled into view, and the result message was announced in-page. 

// Before (AI draft)
alert("Please answer all questions");

// After (implemented)
resultsSection.innerHTML = '<p class="quiz-error-msg" role="alert">Please answer every question...</p>';

4. Example 3: beforeunload warning

Prompt used

“Add a beforeunload warning so if someone starts the quiz and then tries to close or refresh the tab, the browser warns them they might lose their attempt.”

Summary of AI output

It suggested a boolean flag (for example started) set to true when the user first interacts with the quiz (for example on change for a radio), plus window.addEventListener('beforeunload', …) that calls preventDefault() and sets e.returnValue to a non-empty string only when that flag is true, which triggers the native confirmation dialog in supporting browsers.

Changes made to the code

  • After a successful submit (all questions answered and scored), I set started = false so leaving the page no longer shows the warning—without that, the flag stayed true and the prompt still appeared even though the attempt was finished.
  • Left the listener registered globally but gated on the flag, rather than adding/removing listeners on every interaction, to keep the flow simple and match the rest of the quiz script.

Decision and verification

Decision: I accepted the guarded beforeunload design and refined lifecycle handling after submit. Verification: I checked that refresh/navigation prompts appeared only while the attempt was in progress and stopped after a successful submission. 

// Before (AI draft idea)
let started = false;

// After (implemented behaviour)
if (!submitted && isDirty) {
  e.preventDefault();
  e.returnValue = "";
}

5. Example 4: Pass reward via public API

Prompt used

“After a 70% pass, fetch a random quote with fetch from a public JSON API, validate the response shape, and show quote text and author in the results area—no hard-coded inspirational strings.”

Summary of AI output

The model sketched a fetch(url).then(r => r.json()) chain and suggested displaying data.content and data.author. It did not always check HTTP status, Content-Type, or reject oversized strings before inserting into the DOM.

Changes made to the code

  • Moved the call into js/api.js as QuizApi.fetchPassRewardQuote(), with normaliseQuotePayload() that requires a non-empty content string and caps length before use (escapeHtml in quiz.js when rendering).
  • Reject non-JSON responses via the Content-Type header check; return null on network errors so the quiz can show a transparent fallback message instead of a fake quote.

Decision and verification

Decision: I accepted the API integration concept but rejected the unvalidated payload assumption. Verification: I tested pass/fail paths and simulated API failure, confirming real quotes render when valid and a non-fake fallback appears when responses are invalid. 

// Before (AI draft)
const data = await response.json();
show(data.content, data.author);

// After (implemented)
if (!isValidQuotePayload(data)) return null;
resultsSection.innerHTML += renderSafeQuote(data);

6. AI limitation

Mistake: For shuffling, the first snippet suggested sorting with a random comparator: array.sort(() => Math.random() - 0.5) (a pattern that still appears in some draft files). That approach does not produce a uniformly random permutation; some orderings are more likely than others, which is unfair for a quiz.

How I found it: I searched for “shuffle array javascript biased” after noticing odd repeats when refreshing, then read MDN/community notes on why sort with random compare is flawed. I also logged a few dozen shuffles and saw certain indices sticking near the top more often than chance would suggest.

Fix: I implemented Fisher–Yates in the quiz script: copy the array, loop from the end, swap with a random index at or below the current position. That matches the intended behaviour and removed the bias.

7. Critical evaluation

Pros: AI sped up boilerplate (DOM loops, template strings for labels, repetitive CSS) and reminded me of APIs I use infrequently, such as scrollIntoView options. It was useful for structuring try/catch around fetch and for sketching the attempt-history shape stored in localStorage.

Cons: Outputs often assume a single-file demo or a framework I am not using, so IDs, paths, and where code runs (inline vs external script) need careful editing. The shuffle bug is a good example of “looks correct” code that is still wrong under scrutiny.

Reliability and risks: Models can hallucinate library APIs or slightly wrong event names; reliability is only as good as my verification in the browser and DevTools. Risks include copying insecure patterns (for example embedding user input into innerHTML without escaping) or over-trusting generated accessibility markup without testing with keyboard and screen reader basics. For this project, treating AI as a draft assistant and owning the final test pass kept those risks manageable.
