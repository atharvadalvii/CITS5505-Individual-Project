Web basics tutorial

This tutorial is written for beginners and follows the same order real pages are built: first HTML for meaning and accessibility, then CSS for layout and visual design, then JavaScript for interaction. Each section explains both what a concept does and why it matters when you build real interfaces, so you can move beyond memorising definitions and start making good implementation decisions.

HTML Basics

What you will learn: think of HTML as the page's content blueprint. Good HTML does more than make text appear: it tells browsers and assistive technologies what each part means. For example, a real <button> is focusable by keyboard, announces itself correctly to screen readers, and responds to Enter/Space by default. A clickable <div> looks similar but misses those built-in behaviours unless you re-implement them manually.

Why this matters in practice: semantic HTML saves debugging time later. Clear heading order, labels tied to controls, and landmark elements such as <nav>, <main>, and <footer> make pages easier to navigate for both users and developers.

Common beginner mistake: choosing elements by appearance instead of meaning. A heading should be a heading (<h1> to <h6>), a control should be a control (<button>, <input>), and navigation links should sit in a <nav>. If you start with semantic structure, CSS can style it later without breaking accessibility.

Beginner boilerplate (copy to start): this is a minimal HTML page skeleton. It includes encoding, responsive viewport, and links to your CSS and JavaScript files.

How to read it: <!DOCTYPE html> tells the browser to use modern HTML rules. <head> stores page metadata (title, charset, viewport), while <body> contains visible content. The <link> line loads your stylesheet, and the <script> line loads your JavaScript file so behaviour can run after content exists. 

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My First Page</title>
  <link rel="stylesheet" href="styles.css">
</head>
<body>
  <h1>Hello, web!</h1>
  <p>I am learning HTML basics.</p>
  <script src="app.js"></script>
</body>
</html>

CSS Basics

What you will learn: CSS controls how content looks and adapts across screen sizes. A beginner-friendly way to write CSS is to use reusable tokens (custom properties) for color, spacing, and typography. That avoids hard-coding random values in many places and makes global updates easier, such as adding dark mode or adjusting spacing consistently.

Deeper decision-making: when choosing between flex and grid, ask what layout problem you are solving. flex is often best for one-dimensional alignment (row or column), while grid is better for two-dimensional page regions. This reasoning prevents trial-and-error CSS and helps you design predictable layouts.

Practical workflow: define tokens in :root, build component styles with those tokens, then add media queries for responsive behaviour. This keeps your CSS maintainable because design decisions (like spacing or accent colour) are centralised instead of repeated.

Beginner boilerplate (copy to start): this stylesheet gives you basic readable defaults, spacing tokens, and one button style so your page already looks structured.

How to read it: :root defines reusable variables (tokens) like colors and spacing. The body rule sets global defaults (font, background, text color, outer spacing). The .btn class styles a reusable button. This keeps your CSS clean because you can edit token values once instead of updating every component manually. 

:root {
  --bg: #f8f9fb;
  --text: #1f2937;
  --accent: #2563eb;
  --space-4: 16px;
}

body {
  margin: 0;
  padding: var(--space-4);
  font-family: Arial, sans-serif;
  background: var(--bg);
  color: var(--text);
}

.btn {
  padding: 10px 14px;
  border: none;
  background: var(--accent);
  color: #fff;
  cursor: pointer;
}

  --space-4: 16px;

  --accent: #2563eb;
}

  padding: var(--space-4);
  border-radius: var(--radius);
  border: 2px solid var(--accent);
}

.card h2,
.card h3 {
  color: var(--accent);
}

.accent-chip {
  display: inline-block;
  padding: 6px 10px;
  border-radius: 6px;
  background: var(--accent);
  color: #ffffff;
  font-size: 0.875rem;
}

.card-row {
  display: flex;
  gap: var(--space-4);
}

@media (max-width: 700px) {
  .card-row { flex-direction: column; }
}

JavaScript Basics

What you will learn: JavaScript adds behaviour: it reacts to user actions, updates text, validates input, and coordinates UI state. A useful mental model is event -> state -> UI update. First an event happens (click, input, submit), then your code updates state (variables or storage), then the DOM reflects that state so users see the change.

Common beginner pitfall: waiting for DOMContentLoaded avoids reading elements before they exist. This is a common beginner bug: code works sometimes because script timing is inconsistent. Guarding element lookups with null checks also makes scripts robust when markup changes.

Safe update pattern: prefer textContent when inserting user input, keep state in clearly named variables, and centralise rendering in one function. This pattern scales from tiny demos to larger features because display logic stays in one place.

Beginner boilerplate (copy to start): this JavaScript pattern waits for the DOM, safely finds elements, and updates text on click.

How to read it: DOMContentLoaded waits until HTML is parsed before running code. getElementById grabs elements you want to control. The safety check (if (!btn || !out) return;) prevents errors if IDs are missing. Finally, addEventListener('click', ...) reacts to user interaction and updates the page with textContent. 

document.addEventListener('DOMContentLoaded', function () {
  var btn = document.getElementById('helloBtn');
  var out = document.getElementById('helloText');
  if (!btn || !out) return;

  btn.addEventListener('click', function () {
    out.textContent = 'Great! Your JavaScript is working.';
  });
});

Debugging habit: when behaviour fails, check three things in order: (1) did the event fire, (2) did state change, (3) did render run. This sequence maps directly to the event -> state -> UI model and helps you find bugs quickly without guessing. 

// Lightweight trace while learning
console.log('event: input fired');
console.log('state:', state);
console.log('ui:', preview.textContent);
